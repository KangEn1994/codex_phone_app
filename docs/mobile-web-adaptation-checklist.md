## P0 必做

### 1. 移动端运行态识别

目标：

- 让页面知道当前是在移动端壳内运行

建议方式：

- 识别自定义 User-Agent
- 或读取 `client=mobile`

用途：

- 隐藏桌面专属入口
- 按需切换布局

### 2. 隐藏“终端打开”

原因：

- 这是桌面宿主能力
- 移动端没有等价方案

处理方式：

- 在移动态直接不渲染
- 或显示禁用态说明

当前涉及位置：

- 会话卡片操作区
- 当前会话操作区

### 3. 安全区适配

目标：

- 避免内容被状态栏、刘海、底部手势区遮挡

建议改动：

- 为顶层容器补 `env(safe-area-inset-*)`
- 检查底部输入区和顶部标题区

### 4. Cookie 生产配置

目标：

- 确保移动端壳在正式环境下可安全保持登录态

建议改动：

- 按环境切换 `secure`
- 视部署情况检查 `samesite`

## P1 建议做

### 1. 新建会话交互改造

现状问题：

- `cwd` 依赖手输路径
- 手机上输入服务器目录不友好

建议方案：

- 基于现有 `/api/projects` 做可选列表
- 默认带入当前项目

不变部分：

- 后端接口不变
- 提交流程不变

### 2. 布局压缩

现状问题：

- 当前页面信息密度偏高
- 三栏结构在小屏上需要更强切换

建议方向：

- 保持当前逻辑
- 只优化小屏布局和层级
- 优先保证会话列表、消息区、输入区可用

### 3. 加载态和错误态

建议补充：

- 首屏加载中状态
- 接口失败提示
- WebSocket 断线提示
- 空状态文案优化

## P2 可选

### 1. 移动端分享

- 仅在壳应用需要时增加 JS bridge

### 2. 深链

- 支持直接打开某个会话

### 3. 推送

- 需要原生和后端再协作，不在当前最小改造内

## 不建议在当前阶段做的改动

1. 前端框架重构
2. API 协议重设计
3. 前后端分离重做
4. 本地缓存业务状态
5. 为移动端复制一套独立页面

## 开发时的改动原则

1. 能用样式解决的，不改接口
2. 能用运行态判断解决的，不拆逻辑
3. 能复用现有请求方式的，不引入跨域方案
4. 移动端特性尽量收敛在薄兼容层

## 验证顺序

1. 登录是否稳定
2. 会话列表是否可用
3. 消息流和 WebSocket 是否稳定
4. 新建会话是否顺手
5. 安全区和键盘弹起是否正常
