# 移动端壳工程运行时规格

## 目的

这份文档用于约束 Android 和 iOS 壳工程的一期实现方式。

目标不是描述业务功能，而是明确壳工程在运行时必须遵守的规则：

- 入口页面如何加载
- 哪些地址允许留在壳内
- 哪些地址必须跳系统浏览器
- 加载失败如何处理
- 登录态如何保持
- 页面和壳之间如何识别彼此

## 一期结论

Android 和 iOS 都采用：

- 单主容器页
- 单 WebView
- 前端页面内承载全部业务交互
- 原生层只负责容器控制和少量系统能力

一期不做：

- 原生业务页面
- 原生登录页
- 原生消息页
- 原生状态管理

## 运行时总体模型

```text
App Launch
  -> Splash / Loading
  -> Main Web Container
  -> Load BASE_WEB_URL
  -> Web 登录 / 会话 / 消息 / 设置
  -> 原生层处理导航、异常、外链、生命周期
```

## 入口加载规格

### 基础入口

壳工程必须有唯一入口配置：

- `BASE_WEB_URL`

示例：

- `https://codexapp.example.com/`

要求：

1. 必须是 HTTPS
2. 必须指向正式可访问页面
3. 不允许把本地调试地址作为默认发行配置

### 首次加载流程

1. App 启动
2. 显示原生启动页
3. 初始化 WebView
4. 加载 `BASE_WEB_URL`
5. 页面成功加载后隐藏原生 loading
6. 如果加载失败，进入原生错误页

### 页面重载规则

允许以下方式触发重载：

- 用户手动点击“重试”
- 网络恢复后用户重新进入页面
- 壳工程检测到不可恢复错误后执行一次完整 reload

不建议自动无限重试。

## 环境配置规格

壳工程按环境提供以下配置项：

- `BASE_WEB_URL`
- `ALLOWED_HOSTS`
- `EXTERNAL_HOSTS`
- `CUSTOM_USER_AGENT_SUFFIX`
- `ENABLE_WEB_DEBUG`

建议示例：

```text
BASE_WEB_URL=https://codexapp.example.com/
ALLOWED_HOSTS=codexapp.example.com,api.codexapp.example.com
EXTERNAL_HOSTS=openai.com,github.com
CUSTOM_USER_AGENT_SUFFIX=CodexAppMobile/1.0
ENABLE_WEB_DEBUG=false
```

说明：

- `ALLOWED_HOSTS`：允许继续留在 WebView 内部的域名
- `EXTERNAL_HOSTS`：允许跳系统浏览器的域名
- 未出现在任何白名单中的地址，默认拒绝直接打开

## 导航控制规格

### 导航决策原则

任何即将跳转的 URL，都必须经过原生层统一判定。

判定结果只有三种：

1. 在 WebView 内继续打开
2. 转系统浏览器打开
3. 直接拦截并提示

### 允许留在壳内的地址

满足以下条件之一可留在壳内：

1. 与 `BASE_WEB_URL` 同 host
2. 在 `ALLOWED_HOSTS` 列表内
3. 属于业务明确允许的登录跳转地址

### 必须转系统浏览器的地址

满足以下条件之一应转系统浏览器：

1. 在 `EXTERNAL_HOSTS` 列表内
2. 明确为外部文档、帮助页、隐私政策页
3. 属于第三方站点

### 必须拦截的地址

以下情况直接拦截：

1. 明显非法 URL
2. 非 HTTPS 且不是系统明确允许的协议
3. 不在白名单中且也不允许外跳

### 特殊协议处理

以下协议需要单独判断：

- `mailto:`
- `tel:`
- `sms:`

处理原则：

- 如果产品需要，交给系统处理
- 如果当前版本不需要，直接拦截

## 导航矩阵

### 页面内导航

- 同域业务页：WebView 内打开
- 同域 API 请求：页面自身处理，原生层不介入
- 同域 WebSocket：页面自身处理，原生层不介入

### 外链导航

- 白名单外链：系统浏览器打开
- 未知外链：默认拦截

### 新窗口行为

如果页面触发：

- `target="_blank"`
- `window.open`

处理规则：

1. 不创建第二个 WebView
2. 先走统一 URL 判定
3. 同域则复用当前容器打开
4. 外域则转系统浏览器

## 登录态规格

### 一期策略

继续沿用现有 Web 登录态：

- Cookie 登录
- 同域请求
- 同域 WebSocket

原生层不接管：

- 登录表单
- Token 存储
- 用户状态管理

### 保持规则

要求：

1. 用户首次登录成功后，重新进入 App 应尽量保持登录态
2. App 从后台恢复时，不强制刷新到登录页
3. 仅在服务端会话失效时回到 Web 登录页

### 清理规则

以下情况下允许清 Cookie：

1. 用户主动退出登录
2. 用户主动清除应用数据
3. 调试环境手动重置

一期不做“原生层强制登出所有 Web 状态”。

## WebView 行为规格

### 共性要求

1. 启用 JavaScript
2. 允许 Cookie
3. 允许 HTTPS 页面正常缓存静态资源
4. 不注入与业务无关的复杂 JS bridge
5. 控制台日志和网页调试能力仅在开发环境启用

### Android

要求：

1. 返回键优先执行 WebView 回退
2. 仅在无法回退时退出当前容器
3. 支持 HTTPS Cookie
4. 支持 WebSocket

### iOS

要求：

1. 使用 `WKWebView`
2. 使用受控导航委托统一做 URL 判定
3. 支持 HTTPS Cookie
4. 支持 WebSocket

## 页面与壳的识别规格

### 一期推荐

壳工程在 User-Agent 末尾追加：

- `CodexAppMobile`
- 平台信息
- App 版本号

示例：

```text
Mozilla/5.0 ... CodexAppMobile iOS/1.0.0
Mozilla/5.0 ... CodexAppMobile Android/1.0.0
```

### Web 侧用途

Web 页面据此识别：

1. 当前是否运行在壳中
2. 当前平台是 iOS 还是 Android
3. 是否启用移动端 UI 分支

### 一期不建议

一期不建议上来就引入复杂的 JS bridge 协议。

如果只是为了识别运行态，User-Agent 足够。

## 错误页规格

### 需要覆盖的错误类型

1. 首屏加载失败
2. 域名不可达
3. 网络断开
4. 证书错误
5. 被导航策略拦截

### 错误页最小信息

必须展示：

- 错误标题
- 简短说明
- 重试按钮

可选展示：

- 当前目标地址
- 错误码
- 联系支持方式

### 错误页文案原则

不要暴露过多技术细节给普通用户，但调试版可保留更多信息。

## 加载态规格

### 首屏加载

壳层显示原生 loading，直到：

- 页面首次成功提交并渲染
- 或者进入错误页

### 页面内加载

业务页面内部的 loading 仍由 Web 自己管理，原生层不重复覆盖。

## 生命周期规格

### App 切后台

一期策略：

- 不主动销毁 WebView
- 不主动清理 Cookie
- 不主动强制 reload

### App 回前台

一期策略：

- 保持当前页面状态
- 如果页面已失效，由 Web 自身或用户操作触发刷新

### 内存压力

如果系统回收导致 WebView 状态丢失：

- 允许重新加载 `BASE_WEB_URL`
- 不要求恢复到精确滚动位置

## 网络规格

### 必须要求

1. 正式环境仅允许 HTTPS / WSS
2. WebSocket 必须可用
3. 壳工程不得自行改写业务 API 路由

### 不建议

1. 不建议壳工程自行代理 API
2. 不建议壳工程拼接业务请求头干预现有鉴权
3. 不建议本地缓存业务响应用于离线展示

## 日志规格

### 开发环境

允许记录：

- 当前加载 URL
- 导航判定结果
- WebView 错误事件
- 外链跳转事件

### 生产环境

只保留必要埋点：

- 首屏成功/失败
- 重试次数
- 页面崩溃或加载失败

不得记录：

- 用户敏感会话内容
- 消息正文
- Cookie 明文

## 待确认项

以下内容在正式开发前必须确认：

1. 正式域名和测试域名分别是什么
2. 是否存在登录后跳转到其他主域的场景
3. 是否需要支持 `mailto:` / `tel:`
4. iOS 最低版本和 Android 最低版本
5. 是否需要沉浸式启动页
6. 是否要求弱网下自动重连提示

## 开发前建议结论

客户端同学开工前，只要统一以下五件事，后续不会乱：

1. 固定 `BASE_WEB_URL`
2. 固定 `ALLOWED_HOSTS`
3. 固定 User-Agent 追加规则
4. 固定外链跳转规则
5. 固定错误页和重试策略
